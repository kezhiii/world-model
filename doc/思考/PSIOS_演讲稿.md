# PSIOS · 原生物理空间智能操作系统 — 演讲稿

## 一、开场：为什么太空自动驾驶行不通

大家好。今天我要向大家介绍一个关于未来太空智能系统的基础架构设想，我们称之为 PSIOS——原生物理空间智能操作系统。

在正式开始之前，我想请大家先思考一个问题：过去十年，地面的自动驾驶技术发展非常快，Tesla FSD、Waymo、百度 Apollo，已经能够处理感知、预测、规划、控制的完整闭环。那一个非常自然的问题就是——我们能不能直接把地面的自动驾驶技术搬到航天器上，实现卫星自动驾驶？

从表面上看，两者确实很像。汽车和卫星都是运动平台，都需要定位、导航、避障和自主控制。事实上，近年来已经有很多研究开始把世界模型、强化学习、端到端神经网络引入航天领域，希望复刻地面自动驾驶的技术路线。

但是，我今天想和大家讨论一个根本性的观点：这条路行不通。不是技术不够好，而是前提错了。

**地面自动驾驶面对的是一个经验驱动的开放世界。** 行人不可预测，前车司机可能突然变道，道路施工、大雾、暴雨——这些都没有统一的数学模型可以精确描述。所以自动驾驶必须依赖海量数据来学习隐含模式，它学的是"世界本身"。

**而航天器面对的是一个物理驱动的确定性世界。** 轨道运动严格遵循经典力学，两体动力学方程、J2 摄动、大气阻力、太阳辐射压——这些都是经过几十年工程验证的确定性方程。对于轨道传播，误差主要来源于参数精度，而不是环境本身的不确定性。

换句话说，对航天器来说，AI 没有必要重新学习已经被物理规律准确描述的世界。真正需要智能学习的，是那些动力学模型无法覆盖的部分——未知扰动、任务意图理解、多目标决策、风险评估、自主任务规划、多航天器协同。

我提炼一个核心公式：**地面自动驾驶学习的是 Physics Unknown——未知的物理；而太空自动驾驶应当学习的是 Physics Beyond——物理之外的语义和不确定性。** AI 不应该是物理模型的替代者，而应该是物理模型的扩展者。

下面我们来看当前的困境。目前航天自主控制领域存在两条技术路线，但它们彼此完全割裂。

第一条是**模型驱动**路线。它完全建立在轨道动力学基础上，通过最优控制、MPC、轨迹优化做自主导航。优势很明显——可解释、可靠、工程验证充分。但问题也很明显——缺乏学习能力，遇到未知环境就手足无措。

第二条是**数据驱动**路线。用神经网络、强化学习直接学轨迹规划和控制策略。这里有两个致命问题。第一，神经网络往往重新学习已经明确的轨道动力学规律——神经网络花大量计算资源去拟合一个用解析公式就能精确计算的东西，这是巨大的浪费。第二，轨道控制属于 Safety-Critical 系统，任何控制策略都必须满足推进能力、燃料、轨道和安全约束，纯神经网络缺乏显式物理约束，决策过程是黑箱，没法做安全验证。

**所以目前的局面是：模型驱动强调物理规律却没有学习能力，数据驱动强调学习能力却忽略了物理规律——整个领域卡在了这两条路线之间的鸿沟里。**

我们的回答是：不要选边站。我们要做的，是重新定义物理模型和神经网络之间的关系。

这就是我们提出 **Physics-Native Intelligence——原生物理智能**的出发点。

## 二、什么是 Physics-Native Intelligence

大家可能听说过 Physics-Informed Learning——在神经网络的损失函数里加一个物理约束项，让网络预测的结果满足某个偏微分方程。这是一个很好的思路，但它本质上仍然把物理规律当作"外部约束"，训练完之后物理模型就退出了。

**Physics-Native Intelligence 不一样。** 物理规律不是外部的约束条件，而是智能产生过程的内部组成部分。物理模型不是训练阶段的约束，而是运行时持续参与的组件。它和神经网络一起完成推理——神经网络不会覆盖物理模型，而是在物理模型的确定性基础之上，学习未知的知识。最终的智能行为，来自物理推理和数据推理的持续协同。

我们用三个英文词来概括这个三元结构：**Evolution、Cognition、Decision。**

**Evolution——物理模型**负责描述世界如何演化。轨道在未来十分钟会什么样？可达域是哪里？还剩多少推进剂？是否满足安全约束？这些都是基于动力学方程精确计算的，交给物理模型。

**Cognition——神经网络**负责理解世界意味着什么。目标卫星是否正在主动接近我？它是否可能改变轨道？周围有没有碎片碰撞风险？当前通信窗口还能用多久？这些信息无法从动力学方程中直接获得，需要通过学习来形成认知。

**Decision——智能决策**负责确定下一步应该做什么。在物理约束和认知信息的基础上，生成候选行为、评估优劣、选择最优方案。

三者共同参与推理，它不是一个 Physics + AI 的简单叠加，而是三者深度融合的统一体系。

在这个体系下，智能不是一次模型推断的结果，而是一种持续运行的能力。系统运行的过程是一个永不停歇的循环：从当前状态出发，物理模型传播未来；神经网络分析环境语义；系统生成候选行为；物理模型验证可行性；通过则执行，不通过就重新优化。整个过程不断循环，整个任务生命周期内不停止。**智能不是一次计算结果，而是一种持续运行的能力。**

基于这个范式，我们提炼了四项贯穿全局的设计原则：

**第一，Physics First——物理优先。** 所有状态演化必须遵循客观物理规律，任何学习算法不能突破动力学约束。物理模型在所有推理环节中处于最高优先级——它首先确定"什么是可能的"，然后神经网络再在可能空间中寻找"什么是最优的"。

**第二，Responsibility Separation——职责分离。** 物理模型负责世界演化，神经网络负责环境认知。两者职责明确，不相互替代。神经网络不应该学习已经由物理定律描述的内容——这是避免重复学习的关键。

**第三，Continuous Reasoning——持续推理。** 智能来源于持续循环的推理过程，不是单次模型推断。系统不会因为某个任务完成而停止思考，而是在整个任务生命周期内持续进行状态传播、环境认知、策略优化和行为验证。

**第四，Extensibility——可扩展。** 系统能不断增加新的智能能力，而无需改变底层运行机制。新能力以插件形式加入，这要求系统以能力单元而非功能模块来组织。

这里我想特别说明一下，为什么我们的标题里有"操作系统"这四个字。这不是修饰词。PSIOS 之所以是一个操作系统，因为它满足操作系统的核心定义。

传统操作系统管理 CPU、内存、I/O——PSIOS 管理推进剂、电源、通信、计算资源。传统操作系统管理和调度进程——PSIOS 管理和调度 Runtime。传统操作系统通过内核态/用户态保证安全——PSIOS 通过 Physics-First 约束验证保证安全边界不能突破。传统操作系统有文件系统做数据持久化——PSIOS 通过 USS，也就是统一状态空间，做状态持久化。传统操作系统不关机——PSIOS 的智能引擎在整个任务周期内不停止。

在这个意义上，**PNIE，也就是 Physics-Native Intelligence Engine，是 PSIOS 的内核。** PNIE 之于 PSIOS，相当于 Linux Kernel 之于 Linux Distribution。PSIOS 还包括任务接口层、遥测遥信子系统、地面端的数字孪生仿真环境等完整的操作系统生态。

## 三、PNIE 总体架构：Runtime 体系的诞生

现在我们来深入 PNIE 的架构设计。

PNIE 位于任务系统与飞行控制系统之间。上层任务接口说"去接近目标卫星"，下层 GNC 负责执行姿态控制、轨道机动和推力器驱动。PNIE 的职责，是把"去接近目标卫星"这个模糊的任务目标，转化为一系列满足物理约束的自主行为。

传统航天软件是怎么组织的？按功能分成独立的模块——轨道确定、轨迹规划、姿态控制、任务管理，模块之间通过预定义接口交换数据。这套设计在任务简单时没问题，但一旦任务复杂度增加，模块之间就会产生大量耦合。增加一个自主交会能力，可能要同时修改轨道模块、规划模块、控制模块的接口，N 个模块产生 N² 条通信路径。

**PNIE 采用了完全不同的组织方式。** 它不是按功能划分模块，而是按智能能力的产生机制来组织。导航、交会、补给——这些都是不同的能力表现，但它们都依赖同一套底层机制：物理认知、环境理解、行为规划、资源管理。所以 PNIE 为这些共享机制定义了统一的运行单元，我们称之为 **Runtime**。

Runtime 不是一个普通的软件模块。传统的 Module 是"输入→计算→输出→结束"的一次性函数。而 Runtime 是一个具有持久生命周期的能力单元——它在整个任务过程中持续运行，不断更新自己的内部状态，永远不退出。

我们总结一下 Runtime 的四个关键属性：**持续性**——它不是算完就退出的函数；**独立状态管理**——每种 Runtime 有自己维护的状态空间；**松耦合通信**——Runtime 之间不直接调用，而是通过 USS 共享状态；**异步运行**——轨道传播每秒更新，认知推理可能十秒一次，行为规划只在必要时触发，各自独立节拍。

在 PNIE 中，我们定义了四类核心 Runtime：

**第一个是 Physics Runtime。** 它回答的问题是：世界如何演化？它维护航天器及空间环境的物理状态——轨道、姿态、资源消耗、物理约束。它是整个系统可信性的基础。

**第二个是 Neural Runtime。** 它回答的问题是：世界意味着什么？它不学动力学、不预测轨迹、不输出控制量——这些都由其他组件负责。它关注的目标意图、任务语义、环境风险、多目标协同。传统 AI for Aerospace 的神经网络是"物理替代者"，PNIE 的 Neural Runtime 是"物理增强者"。

**第三个是 Behavior Runtime。** 它回答的问题是：下一步做什么？它是整个系统的决策中枢——综合 Physics 提供的事实和 Neural 提供的认知，生成当前最优行为。

**第四个是 Memory Runtime。** 它回答的问题是：过去教会我们什么？这是跨越多个任务周期的长期知识积累层，让系统真正实现"越飞越聪明"。

这四类 Runtime 没有主从关系。它们都围绕一个统一的状态空间——USS——独立运行。Physics Runtime 更新了轨道状态，Neural Runtime 自动读取到；Neural Runtime 更新了风险评估，Behavior Runtime 在下一周期立即感知到。整个系统以状态驱动的方式协同，而不是以模块调用的方式耦合。

最后我想强调一个工程层面的点：PNIE 的计算量有多大？Physics Runtime 的轨道传播本来就是传统航天任务中一直在做的事情，ARM M7 级别的处理器就能跑。Neural Runtime 的认知推理需要 NPU 或者低功耗 GPU，但更新频率很低，0.1 到 1 赫兹就够了。Behavior Runtime 和 Memory Runtime 的计算量更低。星载平台比如 Xilinx Versal 或者 NVIDIA Jetson Orin 的抗辐射版本，已经足够支持这个负载。**关键不是算力绝对值，而是 Physics Runtime 承担了计算量最大的部分，Neural Runtime 仅处理认知推理——这比纯神经网络方案的计算需求低得多。**

## 四、Physics Runtime：数字物理世界层

现在我们来深入 Physics Runtime 的具体设计。

在传统航天软件里，轨道传播是一个算法工具——你调用它，它给你算一段未来的轨道，然后它就退出了。但在 PNIE 中，物理模型不再是一次性的工具，它被提升为一个持续运行的 Runtime。它始终维护航天器的完整物理状态，随着时间不断演化，为其他 Runtime 提供唯一可信的物理事实来源。

**我们称之为 Digital Physics Layer——数字物理世界层。** 这相当于把航天器的物理世界从离散的计算工具提升为持续运行的基础设施。

Physics Runtime 有五大核心职责：

**第一，动力学传播。** 从最简的两体模型，到含 J2、J3 摄动的模型，再到完整力模型加数值积分的高精度轨道传播，以及姿态动力学——Physics Runtime 不限制具体用什么传播模型，但要求所有传播结果有统一的数据接口。这保证了系统未来更换传播算法时，对外接口不变。

**第二，约束维护。** 任何航天器都受大量约束——推进剂还能用多少？推力器最大推力是多少？地面站过境窗口什么时候？太阳角是否在散热面的允许范围内？姿态机动有没有禁区方向？电池还剩多少？传统系统中这些约束分散在各个模块中单独计算。PNIE 中 Physics Runtime 持续维护所有这些约束，其他 Runtime 直接从 USS 读取，不需要重复计算。

**第三，可达域分析。** 系统不仅要知道"现在在哪"，更要知道"未来能去哪"。给定当前推进剂余量，哪些轨道是可以到达的？最快需要多长时间？有没有不能去的区域？这些信息是 Behavior Runtime 做行为规划时最重要的输入。

**第四，资源演化。** 推进剂消耗、电池荷电状态、太阳板输入功率、热平衡、执行机构寿命——这些都是物理量，有明确的数学规律，由 Physics Runtime 统一维护。

**第五，也是最关键的——物理一致性验证。** Physics Runtime 是整个 PNIE 的 **Safety Guardian，安全守护角色。** 任何 Behavior Runtime 输出的候选行为，都必须通过 Physics Runtime 的四项验证：动力学是否一致？工程约束是否满足？安全边界是否满足？资源是否可行？只有全部通过验证的行为，才能进入 GNC 执行。

这里我想强调 Physics Runtime 和 Neural Runtime 之间的边界。**Physics Runtime 不学习、不预测任务语义、不输出行为策略。Neural Runtime 不负责轨道传播、不维护资源状态、不修改动力学模型。** 两者的唯一联系是通过 USS 共享状态。Physics 提供客观事实，Neural 在事实基础上形成认知。这个边界是 PNIE 中最重要的一条线。

## 五、Neural Runtime：从"状态预测"到"认知建模"

下面我们来谈 Neural Runtime——这是 PNIE 与传统 AI for Aerospace 最大的区别所在。

传统 AI for Aerospace 的工作，几乎无一例外地让神经网络去学动力学、学轨迹、学控制策略。它的前提假设是：神经网络是环境的替代者——我把数据给它，它学会这个世界怎么运转，然后直接给我控制指令。

但我们已经反复论证过：在航天领域，这个假设是错的。轨道动力学已经有成熟的解析模型，不需要重新学习。真正需要的，是理解物理状态背后的语义。

我给大家举一个具体的例子。假设有两颗卫星，距离 5 公里，接近速度 0.5 米每秒，位置夹角 15 度。从纯动力学角度看，这两个场景完全一样——相同的相对距离、相同的速度、相同的几何构型。

但它们可能有完全不同的含义。**场景 A**：目标卫星正在按计划执行自主交会任务，主动向我靠近——我应该保持当前交会策略。**场景 B**：目标卫星轨道发生了异常漂移，正在无控地向我逼近——我应该立即进入避碰模式。

仅靠轨道状态——位置、速度——无法区分 A 和 B。你需要看历史轨迹，目标是不是在持续主动接近？你需要看通信信号，目标是不是在发送交会确认？你需要做异常检测，目标的运动是不是偏离了历史模式？

**这就是 Neural Runtime 的职责：它不关注"状态是什么"，它关注"状态意味着什么"。**

所以 Neural Runtime 学的东西和传统方法完全不同。它不学动力学——那是 Physics Runtime 的活。它不学控制策略——那是 Behavior Runtime 加 GNC 的活。它不学完整轨迹——那是 Physics 传播加 Planner 优化的事。**它学的是目标行为意图、任务语义理解、环境风险评估、多目标协同决策、长期经验积累。** Neural Runtime 的输出不是轨迹，不是控制量，而是 Cognitive Information——认知信息。

Neural Runtime 持续维护一个 Space Cognitive State，包含五个维度：环境理解——目标分布、碎片区域、电磁环境；任务理解——当前处于什么阶段、完成了多少、优先级如何；意图估计——目标的行为意图、未来动作趋势；风险评估——碰撞概率、资源风险、任务风险；经验上下文——历史上类似场景怎么处理的，哪种策略效果好。

这里有一个具体的运作示例。**Physics Runtime 告诉系统"目标未来十分钟的位置坐标是 X, Y, Z"。**这是在回答"事实是什么"。**Neural Runtime 判断"该目标正在主动接近，交会意图明确，当前风险等级低，建议继续当前交会策略"。**这是在回答"事实意味着什么"。两种信息合在一起，才构成完整的空间智能。

Neural Runtime 的另一个关键特性是**持续学习**。它不采用地面训练好后星上部署的固定模式——太空环境的长期变化、新型目标的出现、意外事件的积累，都要求系统能够在线更新自己的认知模型。更新来自三个方面：任务反馈——当前策略有效吗？环境变化——出现新的碎片云区域？多智能体协同——其他航天器分享的经验？需要强调的是，**持续学习不会改变 Physics Runtime 中的动力学模型，物理规律是不变的，被更新的只有认知模型。**

## 六、Memory Runtime：让系统"越飞越聪明"

现在我来介绍四类 Runtime 中可能最容易被忽略的一个：Memory Runtime。

大家可能问：Neural Runtime 已经有了经验学习的能力，为什么还需要一个专门的 Memory Runtime？区别在于：**Neural Runtime 的 Experience Context 是"工作记忆"——服务于当前任务周期内的认知推理；Memory Runtime 是"长期记忆"——跨越多个任务、多个轨道周期的知识积累和复用。**

举个例子。Neural Runtime 在分析当前目标时，可能会问"这个目标的意图是什么？"Memory Runtime 给出的回答是不同的——它会检索历史上所有类似目标的行为模式，告诉 Neural Runtime"据统计，这个轨道高度上的 GEO 通信卫星，90% 在交会阶段保持位置保持行为，历史异常概率低于 5%"。这是 Neural Runtime 自己的当前观测分析无法提供的先验信息。

Memory Runtime 维护五类长期知识：**任务历史**——每个已完成任务的完整轨迹、决策点、结果；**目标档案**——已知目标的轨道特征、行为模式、交互记录；**策略知识**——不同场景下哪种接近策略最有效，以知识图谱形式存储；**环境知识**——不同轨道区域的特征，碎片密度、通信质量、光照条件；**经验案例**——具有代表性的成功和失败案例及其完整上下文。

当 Behavior Runtime 需要做决策时，Memory Runtime 提供两方面的支持。第一，通过上下文相似检索——"给我找历史上和当前状态最相似的 3 个场景及其处理方式"，用向量嵌入加相似度搜索实现。第二，通过结构化查询——"在过去 30 天内，500 公里太阳同步轨道上发生过多少次碎片近距离交会事件？"

**Memory Runtime 是 PNIE 实现"越飞越聪明"的关键机制。** 它持续积累，从不删除，除非知识被证伪或过时。任务结束后，B 卫星的周期性推力器脉冲行为被记录入库——下次遇到类似轨道特征的卫星，系统不用重新判断，直接检索即可识别。

## 七、USS 和 USR：统一状态空间

现在我要讲整个 PNIE 的中枢——**USS，Unified State Space，统一状态空间。**

前面的架构中有四类 Runtime，它们各自维护不同层次的状态。Physics Runtime 有物理状态，Neural Runtime 有认知状态，Behavior Runtime 有任务状态和行为状态，Memory Runtime 有长期知识。问题是：如果这些状态之间无法统一表达，整个系统就无法形成一致的推理。

比如传统航天控制系统，状态就是一个动力学向量——位置、速度、姿态、角速度。这能支撑轨道控制，但支撑不了"目标可能在执行交会任务"这样的高层认知。而数据驱动方法的神经网络隐状态，虽然能存一些历史信息，但缺乏明确的物理意义，没法做安全审计。

**USS 要解决的核心问题就是：让物理状态和认知状态在同一个框架下表达，共同参与系统推理。**

USS 包含四种基础状态。**Physical State**：客观物理世界——轨道在哪、速度多少、姿态如何、环境条件是什么。由 Physics Runtime 维护。**Cognitive State**：系统对环境的理解——目标意图、碰撞风险、任务语义。由 Neural Runtime 和 Memory Runtime 维护。**Mission State**：当前任务状态——目标是什么、进行到哪一步了、优先级是否变化。由 Behavior Runtime 维护。**Resource State**：航天器自身资源——推进剂、电源、热控、通信、计算。由 Physics Runtime 维护。

但这四种状态不是独立存放的数据库表——它们之间存在持续的动态耦合。轨道机动 → 推进剂减少，电池SOC下降 → 这是 Physical State 驱动 Resource State。系统识别到碰撞风险 → 调整任务优先级 → 这是 Cognitive State 影响 Mission State。新任务目标 → 改变轨迹规划 → 影响未来轨道演化 → 这是 Mission State 反作用于 Physical State。推进剂不足 → 限制可执行行为的范围 → 这是 Resource State 限缩 Mission State。

**整个智能系统的推理过程，就是不断更新这个动态耦合的状态空间的过程。**

USS 是抽象模型，是我们对状态的建模。**USR——Unified Space Representation——是 USS 的运行时实现。** 它在软件中承担具体职责：唯一状态源——一个状态只能在一个地方存；并发控制——多个 Runtime 同时读写时的状态一致性；版本管理——每次更新都产生版本号，支持追溯和回滚；统一访问接口——所有 Runtime 用相同的 API 读写，不管访问的是轨道还是认知；事件通知——关键状态变更时主动通知相关 Runtime；以及状态追溯——每一项状态记录来源 Runtime、更新时间、可信度分数，为安全审计提供能力。

USR 有五项设计原则。**唯一状态源**——系统任意时刻只存在一个统一状态，不允许 Runtime 维护独立的副本。**状态共享而非模块通信**——Runtime 之间不直接调用对方。Behavior Runtime 不直接调用 Physics Runtime 取轨道——它从 USR 读。**异步运行**——不同 Runtime 不同频率，USR 不要求统一时钟。**状态可追溯**——谁改的、什么时候改的、可信度多少，全记录。**统一访问接口**——`Read(StateType, Query) → StateValue`，`Write(StateType, Key, Value, Metadata) → Version`。

**由于所有 Runtime 都只与 USR 耦合，整个系统的耦合度从传统模块架构的 O(N²) 降到了 O(N)。** 这是一个非常重要的架构优化——每种 Runtime 只与一个接口耦合，新增一种能力只需要注册新的 Capability，不需要修改任何已有组件。

说到 Capability，这里我要强调一个关键的设计：**行为生成"与"能力执行"的分离。** Behavior Runtime 负责回答"做什么"——"自主接近"还是"保持等待"？Capability Library 负责回答"怎么做"——Navigation、Docking、Refueling 的具体实现。Behavior Runtime 告诉系统"现在应该自主接近"，但它不关心接近是怎么做到的——是 EKF 还是粒子滤波、是 Lambert 求解器还是连续推力优化，它都不需要知道。它只调用 `Capability.Navigate(target, constraints)`，由 Capability Library 内部完成具体计算。这种设计保证了能力模块的可替换性和可扩展性。

## 八、Behavior Runtime：行为生成机制

现在我们把目光移到决策中枢——Behavior Runtime。

Behavior Runtime 的输出不是控制指令，那是 GNC 的职责。它输出的是**行为——Behavior。** 行为是一种高于控制、低于任务的智能表达。让我用一个具体的例子说明这四层关系。

**任务层**说的是"对目标卫星完成推进剂补给"。**行为层**说的是"接近目标 → 姿态对准 → 对接 → 传输推进剂 → 安全撤离"。**轨迹层**说的是"从当前轨道经过 3 次机动到达对接点，第 1 次 ΔV=6 m/s"。**控制层**说的是"打开推力器 1，推力 50%，持续 120 秒"。

Behavior Runtime 定位在行为层。它是连接高层任务目标和底层控制系统的唯一桥梁。

行为生成遵循一个四阶段的持续循环，每个阶段都在更新和反馈。

**阶段一，状态感知。** Behavior Runtime 从 USS 读取完整的统一状态。物理状态——我在哪里？速度多少？能到达哪里？认知状态——目标在做什么？环境有风险吗？任务状态——我现在处于什么阶段？还需要完成什么？资源状态——我还有多少推进剂？电力够不够？

**阶段二，行为推理。** 基于这些状态回答四个问题：当前最重要的目标是什么？当前是否存在风险？当前资源是否支持继续执行？是否需要切换行为？推理结果不是控制量，而是新的行为状态——例如由"轨道保持"切换为"主动接近"。

**阶段三，能力选择。** 行为确定后，从 Capability Library 中选择对应能力。"自主接近" → Navigation Capability，"姿态对准" → Relative Attitude Capability。

**阶段四，行为反馈。** Capability 执行完成后，新的状态写回 USR。下一个推理周期，Behavior Runtime 重新从 USS 读取，评估是否需要调整行为。

**这四个阶段永不停止。不存在"决策结束"这个时刻。**

为了处理复杂航天任务的多阶段特性，我们采用了**分层行为组织**。**战略行为**关注数小时甚至数天的长期目标——完成整个补给任务、维护编队。**战术行为**关注当前子任务的执行——接近、对接、传输。**执行行为**关注具体能力调用和结果监测。三层分离使 Behavior Runtime 能在不同时间尺度上完成决策，不会把长期规划与短期控制混在一起。这与人类驾驶员的决策逻辑一致——先决定变道，再判断时机，最后打方向盘。

最关键的一点是**行为切换机制**。传统系统的行为切换通常基于硬编码的阈值——"距离小于 100 米时进入对接模式"。但在真实太空任务中，阈值驱动的状态机是不可靠的。如果目标突然发生异常机动，系统不能因为还没到 100 米就继续接近。

PNIE 的行为切换不依赖固定阈值。每一步切换都由 USS 的多源联合条件决定——物理状态满足下一阶段的动力学条件吗？认知状态确认环境安全吗？资源状态支持吗？历史经验对这次迁移有什么建议吗？五个维度共同满足才触发迁移。

**如果途中 Neural Runtime 检测到目标异常机动，系统可以从"自主接近"立即切回"安全撤离"——不是靠阈值，是靠对全局状态的持续评估。**

## 九、Behavior Graph 和 Physics–Neural Co-Reasoning

接下来我要介绍两个更深入的机制：Behavior Graph——行为组织模型，和 Physics–Neural Co-Reasoning——物理与神经协同推理。

先看 **Behavior Graph。** 对于复杂任务，行为不是孤立的一个动作，而是由多个阶段按照一定逻辑连续演化形成的。Behavior Graph 不是有限状态机，不是预定义的任务脚本，它是 USS 上的一个行为组织模型——描述系统当前处于什么状态，以及如何迁移到下一个状态。

Behavior Graph 由 Behavior Node 和 Behavior Transition 组成。每个 Node 是一个具有明确语义的行为状态。比如"自主接近"节点，它定义了行为目标——把距离缩短到 200 米以内；激活条件——目标已识别并且交会窗口可用；完成条件——距离小于 200 米并且速度匹配；推荐能力——Navigation Capability；安全约束——最小安全距离 50 米，最大接近速度 1 米每秒。

Behavior Transition 则描述两个节点之间如何迁移。迁移条件不是单维度阈值，而是 USS 五个维度的联合判断。物理状态是否满足？认知状态是否确认安全？任务状态是否推进？资源是否支持？历史经验有无负面信号？**只有当所有维度共同满足时，系统才触发迁移。**

Behavior Graph 在整个任务期间持续运行：获取 USS、评估当前节点、判断迁移条件、更新当前节点、调用对应 Capability、等待下一周期。不存在一次性规划后就长期执行的情况。

接下来是 **Physics–Neural Co-Reasoning。** 这是 PNIE 最体现"融合"理念的机制。

我们已经看到 Physics Runtime 回答"世界如何演化"，Neural Runtime 回答"世界意味着什么"。Co-Reasoning 将两者的推理统一到同一个循环中，并加上 Behavior Runtime 的决策，构成一个五阶段闭环。

**阶段一，状态获取。** Behavior Runtime 从 USS 读取完整的统一状态。

**阶段二，物理推理。** Physics Runtime 预测未来——如果什么都不改变，轨道如何演化？相对运动如何变化？能源消耗多少？安全约束是否始终满足？这些预测是确定性的、完全可解释的，因为它来自动力学方程。

**阶段三，认知推理。** Neural Runtime 在物理预测的基础上进行认知分析——目标可能的行为变化、环境风险的趋势、任务成功的概率、置信度估计。这些预测是概率性的，依赖于学习和经验。

**阶段四，行为推理与规划。** Behavior Runtime 综合前两阶段的输出，生成候选行为、用物理约束做筛选、在任务完成度、安全性、资源消耗、长期收益、鲁棒性五个维度上综合评估，选择最优行为。

**阶段五，执行与状态更新。** Capability 执行行为，新的轨道、资源、任务状态写回 USR。下一周期从阶段一开始。

Co-Reasoning 包含一个非常重要的双预测机制。**物理预测**描述的是"未来会发生什么"——精确的、确定性的。**认知预测**描述的是"未来意味着什么"——概率性的、需要学习获取的。Behavior Runtime 在两套预测共同作用下形成最终行为，而不是单独依赖任何一方。

Co-Reasoning 还要求三个维度的一致性。**物理一致性**是硬约束——违反动力学方程的行为直接拒绝。**任务一致性**——技术上可行但偏离任务目标的行为不采纳。**认知一致性**——如果 Neural Runtime 判断目标有异常机动风险，Behavior Runtime 不能继续输出接近行为。

最后我想做个对比。近年来大语言模型的 Agent 通常是"观察→推理→行动"三阶段。**PNIE 的推理循环是"观察→物理推理→认知推理→行为规划→能力执行→状态更新"，多了一个独立的物理推理层。** 而且 LLM Agent 的推理基础是文本，物理约束靠 prompt 实现；PNIE 的推理基础是动力学方程，物理约束是内置的硬保证。PNIE 是面向物理世界的 Agent，不是面向文本世界的 Agent。

## 十、Behavior Candidate Modeling：行为建模与模型选型

现在我们要解决一个更底层的问题：**行为该如何表示？**

Behavior Planner 要在当前状态下寻找最优行为，但如果行为的表示方式本身就不合理，不管后面用 Transformer、Diffusion 还是世界模型，都只能在错误的表示空间里学习。

传统方式几乎毫无例外地选择三种表示：控制量、轨迹、行为标签。

**控制量**——推力器推力、姿态力矩——不行。这直接对应执行机构指令，属于 GNC 的职责。Behavior Planner 应该关注"做什么"，而非"怎么控制"。

**轨迹**——未来位置序列——在航天领域也不行。因为航天器的轨迹由轨道动力学决定，不需要神经网络去学习。给定初始状态和控制约束，物理模型就能精确计算未来轨迹。让神经网络重新学一遍，不仅浪费算力，而且无法保证物理一致性。

**行为标签**——Approach, Wait, Retreat, Avoid——太过粗糙。真实太空任务的行为包含大量连续参数：接近到什么距离？在多长时间内完成？允许消耗多少推进剂？风险偏好是保守还是激进？四个离散标签完全无法表达这些维度。

所以 PNIE 提出了一个全新的统一表示：**Behavior Intent——行为意图。**

Behavior Intent 不是控制量，不是轨迹，而是对未来行为目标的参数化描述。它包含六个参数：**Type**——行为类型，Approach 或 Hold 或 Retreat；**Goal**——行为目标，包括目标 ID、目标距离等连续量；**Constraint**——约束条件，最小安全距离、最大相对速度；**TimeWindow**——时间窗口，最早开始时间和最晚结束时间；**RiskPreference**——风险偏好，Conservative、Balanced 还是 Aggressive；**Budget**——资源预算，ΔV 配额、推进剂配额。

关键在于，Behavior Intent 描述的是行为目标，而不是达到目标的具体路径。之后的流程是：**Physics Runtime 根据 Intent 推导未来物理状态 → Behavior Planner 评估优劣 → Trajectory Planner 生成可执行轨迹。** 这彻底解耦了行为层和控制层。

Candidate Behavior Generator 需要满足四个条件：一对多生成——输出多个候选而不是单一决策；连续参数——处理距离、时间等连续量；上下文感知——结合 USS 中的历史经验和当前环境；Physics Runtime 协同——在物理约束空间内生成，而非独立决策。

那么我们用什么 AI 模型来做这个 Generator？我分析了四种主流模型。

**Transformer** 上下文建模能力强，成熟高效，适合做第一代系统的 Behavior Intent Generator——但它本质是判别式模型，不擅长描述多模态不确定性。

**Diffusion Model** 擅长学习复杂分布，天然支持一对多生成——但推理需要多步去噪，实时性不足。更适合做离线候选生成加在线微调。

**Reinforcement Learning** 能直接学长期收益——但奖励函数怎么设计？安全、燃料、时间，这三者如何统一量化？而且策略黑箱，航天系统没法直接用。

**World Model**——Dreamer 这种——内部仿真能力很强，但它的状态转移完全由神经网络学习，不考虑已有的物理规律。对航天任务来说，这既没有必要性，也降低了可信度。

所以我们的长期目标是：**Physics-guided World Model。** 它的状态转移公式是 $s_{t+1} = f_{physics}(s_t, a_t) + f_{neural}(s_t, a_t, c_t)$。物理模型负责确定性演化，神经网络只学残差——未知扰动、执行器误差、未建模动力学。这相比纯黑盒的世界模型，数据需求大幅降低，可解释性大幅提高，而且 Physics Runtime 保证了预测不会违反任何物理规律。

但是在第一代系统中，我们的工程建议更为务实：**Physics Runtime 做轨道传播和约束预测 → Transformer 做 Behavior Intent 生成 → Behavior Planner 做候选筛选和多维评估 → Trajectory Planner 做轨迹生成。** Transformer 不是 PNIE 的最终形态，但它是当前工程最成熟的行为生成工具。未来逐步演进到 Physics-guided World Model——这正是可扩展原则的体现。

## 十一、场景演练：一次完整的自主交会与补给

现在，让我用一个具体任务，把前面讲的所有组件串起来，展示 PNIE 实际如何运作。

**任务设定：** 服务星 A，在 500 公里太阳同步轨道上。目标星 B，同轨道前方约 50 公里，需要完成推进剂补给。A 的剩余推进剂 2 公斤，最小安全距离 100 米，最大接近速度 2 米每秒。通信窗口 8 分钟后开始，持续 10 分钟。

系统启动。所有 Runtime 同时进入运行状态。Physics Runtime 加载两星轨道参数，开始持续传播。Neural Runtime 加载预训练认知模型。Memory Runtime 加载目标 B 的历史档案——之前有没有交互过？B 的轨道行为特征是什么？Behavior Runtime 读入 USS，确定初始行为。

USS 的初始状态是：物理层面，两星相对距离 50 公里，速度差接近零。认知层面，目标 B 的意图被判断为"位置保持"，置信度 85%。任务层面，目标是交会和补给，阶段是初始化。资源层面，2 公斤推进剂，电力 95%。

Behavior Runtime 推理：当前阶段 init，行为应为"轨道转移"——这是 Behavior Graph 的初始节点。调用 Orbit Transfer Capability。

此时 Physics Runtime 计算可达域：2 公斤推进剂可以提供的总 ΔV 约 18 米每秒。Trajectory Planner 计算霍曼转移路径：两次机动，总 ΔV 约 12 米每秒，剩余裕量充足。Physics Runtime 验证：动力学一致，推进剂充足，安全距离满足。全部通过。GNC 执行第一次机动。

三十分钟后，机动完成。A 接近 B 约 5 公里。USS 更新：物理状态更新为相对距离 5 公里。认知状态更新——B 的意图仍为位置保持。任务阶段推进。

Behavior Graph 评估：当前"轨道转移"节点的完成条件已满足——已经进入 10 公里范围。下一个"自主接近"节点的激活条件也满足——目标稳定、交会窗口可用。系统触发行为迁移。

Neural Runtime 做认知验证：分析 B 的历史轨道数据，确认 B 长期保持稳定，异常概率低于 5%。Memory Runtime 检索到类似轨道上的 3 次历史补给任务都成功完成。认知一致性确认——没有理由不继续。Navigation Capability 开始执行接近。

又过了三十分钟，A 距离 B 约 500 米。**这时发生了关键转折。**

Neural Runtime 检测到 B 的轨道出现微小漂移——Δv 约 0.05 米每秒，非常小。意图估计重新评估，"位置保持"置信度从 85% 跌至 60%，"未知机动"置信度升至 30%。风险评估：B 的不确定性增大，继续接近的风险中等。**认知一致性检查触发——Neural Runtime 认为存在异常机动风险，当前行为"自主接近"需要重新评估。**

Behavior Runtime 现在面临一个真正的决策点。Cognitive State 显示中等风险，Resource State 显示推进剂还够——还剩 1.5 公斤——Mission State 显示任务还没完成。

Behavior Planner 生成三个候选 Intent。**Intent 一：Hold——**维持当前距离 500 米，等判断清楚再行动，风险保守。**Intent 二：Approach——**继续接近到 200 米进入对接准备，风险均衡。**Intent 三：Retreat——**返回 5 公里安全距离，风险保守。

Physics-First 筛选：三个候选都满足物理约束，全部保留。多维综合评估：Intent 一的任务完成度低但安全性高，总分 6.0。Intent 二的任务完成度高但安全性中等，总分 6.3。Intent 三没有任务进展但绝对安全，总分 6.3。系统选择了 Intent 二——继续接近，但认知状态标记为"高度监测 B"。

又过了三十分钟，B 的轨道漂移消失了——Neural Runtime 的意图置信度回升到 90%。原来刚才的漂移是 B 的正常热控推力器周期性脉冲，不是异常机动。

继续接近、姿态对准、对接、补给执行、安全撤离。

总共三小时，任务完成。Memory Runtime 将这次经验写入长期知识库：成功案例加一。B 的轨道行为档案更新——确认存在周期性推力器脉冲，周期约一小时。下次遇到 B 或者类似特征的卫星，系统不需要重新评估三十多分钟，直接检索就能知道"这个漂移是正常的"。

**这次演练展示了 PNIE 最核心的五个特征。** 第一，持续推理——三小时内从未停止，每个决策周期都在重新评估。第二，Physics First——B 只要有一点轨道变化，Physics Runtime 立即更新物理事实。第三，认知驱动的行为调整——Neural Runtime 检测异常后，不是盲目继续接近，而是触发重新评估，生成三个候选综合评分。第四，松耦合——期间 Neural Runtime 更新意图模型、Memory Runtime 更新档案，都不影响 Physics Runtime 的轨道传播。第五，经验积累——任务结束后 B 的新特征被永久记录，下次可快速识别。

## 十二、总结与展望

最后，我想从三个层面来总结 PNIE 的核心贡献。

**在范式层面，** 我们提出了 Physics-Native Intelligence。它和 Physics-Informed Learning 最大的区别是：物理模型不是在训练时约束神经网络，而是在运行时和神经网络平等参与推理。物理模型负责 Evolution，神经网络负责 Cognition，智能决策负责 Decision——三者协同，不相互替代。

**在架构层面，** 我们提出了以 Runtime 为基本单元的 PNIE 架构。四种 Runtime——Physics、Neural、Behavior、Memory——围绕 USS 持续协同运行。整个架构是状态驱动的，不是模块驱动的。物理模型不是被神经网络调用的工具，而是与神经网络平等的运行时组件。系统耦合从 O(N²) 降至 O(N)。可扩展——新增能力只需注册新的 Capability。

**在方法层面，** 我们提出了 Behavior Intent 作为自主规划的统一建模对象。行为从控制量和轨迹中解耦出来，通过"候选生成 → 物理约束筛选 → 多维综合评估"三阶段机制实现 Physics-First 规划。长期目标是 Physics-guided World Model——状态转移由物理模型加神经网络残差共同完成。

PNIE 的设计吸收了多个领域的成熟思想。自动驾驶的 PPPC 分层架构——我们增加了独立的物理推理层。ROS 中间件的松耦合通信——我们将其升级为状态驱动。World Model 的内部仿真机制——我们把物理模型从黑盒中拆解出来。操作系统的进程调度和资源管理——我们把 Runtime 对应为进程，Physics-First 对应为安全内核态。

展望未来，我划分了三个时间阶段。**近期，一到两年**，在数字孪生环境中实现第一代 PNIE 原型，Physics Runtime 加 Transformer Behavior Intent Generator 加 Behavior Planner，验证自主交会和碰撞规避。**中期，三到五年**，建立 Physics-guided World Model 的原型，把物理推理和神经网络推理从串行协同演进到联合建模。**长期，五到十年**，PNIE 从单个航天器扩展到多航天器分布式网络——每颗卫星运行本地 PNIE 实例，通过 USS 同步机制共享物理环境和任务知识。从一颗卫星的自动驾驶，演进为整个空间基础设施的自主运行。

人工智能正在把航天技术推入自主智能时代。但未来真正的空间具身智能，不应仅依赖越来越大的神经网络，也不能完全依赖传统物理模型，而应该建立在二者深度融合的基础之上。

这就是 PSIOS 和 PNIE 想要做的事情。
