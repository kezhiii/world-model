# Physics-Native Space Intelligence Operating System（PSIOS）

## 第一章 重新定义太空自动驾驶——迈向 Physics-Native Space Intelligence

### 1.1 从自动驾驶汽车到太空航天器

以 Tesla FSD、Waymo、Apollo 等为代表的自动驾驶技术快速发展，推动了人工智能从感知识别迈向自主决策与自主控制。现代自动驾驶集环境感知、世界建模、行为预测、路径规划与车辆控制于一体，通常采用"感知—预测—规划—控制（PPPC）"架构。

一个自然的问题是：能否直接将自动驾驶汽车的技术体系迁移到航天器？

从表面看，两者都需要完成自主定位、导航、决策和控制，都在动态环境下保证运行安全。近年来越来越多的研究开始借鉴世界模型、强化学习以及端到端神经网络等技术开展航天器自主导航。

然而，这种直接迁移存在一个根本性的局限：**自动驾驶汽车面对的是经验驱动的开放世界；航天器运行的是物理驱动的确定性世界。** 两类系统智能产生的基础完全不同。

### 1.2 太空自动驾驶与地面自动驾驶的本质区别

自动驾驶汽车面对的高度不确定性——行人不可预测、驾驶员行为随机、道路施工天气变化难以统一建模——决定了它必须大规模依赖数据驱动学习。它学的是"世界本身"。

航天器面对的空间环境截然不同。轨道运动严格遵循经典力学与轨道动力学，状态演化满足确定性方程：

- 两体动力学
- 地球非球形引力（J2/J3）
- 大气阻力
- 太阳辐射压
- 第三天体摄动

这些模型经过几十年工程验证。AI 没有必要重新学习已经被物理规律准确描述的世界。真正需要智能学习的是动力学模型无法完整描述的部分：未知扰动、任务意图理解、多目标决策、风险评估、自主任务规划、多航天器协同。

因此：**自动驾驶汽车学习的是 Physics Unknown，太空自动驾驶学习的是 Physics Beyond。AI 不应该替代物理模型，而应该扩展物理模型。**

### 1.3 当前研究存在的问题

目前存在两类技术路线：

**模型驱动方法**完全建立在轨道动力学基础上，通过最优控制、MPC、轨迹优化等技术完成自主导航。优势在于可解释性和可靠性，但缺乏自主学习能力。

**数据驱动方法**利用神经网络、强化学习等方法直接学习轨迹规划或控制策略。但存在两个问题：

1. 神经网络往往重新学习了已经明确的轨道动力学规律，增加了训练成本，降低了可解释性
2. 纯神经网络难以保证航天系统要求的安全性与可验证性

模型驱动与数据驱动之间仍然存在割裂：模型驱动强调物理规律却缺乏学习能力，数据驱动强调学习能力却忽略了物理规律。这是当前太空智能系统发展的核心瓶颈。

### 1.4 本文提出的观点：Physics-Native Intelligence

**Physics-Native Intelligence** 不是传统意义上的 Physics-Informed Learning（在损失函数中加入动力学约束），而是将轨道动力学作为整个智能系统的原生组成部分（Native Component）。

在该理念下：
- **物理模型**负责描述世界如何演化（Evolution）
- **神经网络**负责理解世界意味着什么（Cognition）
- **智能决策**负责确定下一步应该做什么（Decision）

三者共同参与系统推理，而不是彼此替代。未来的太空自动驾驶不应建立在"Physics + AI"的简单叠加之上，而应建立在物理模型与智能认知深度融合的统一体系之上。

基于这一思想，本文提出 **Physics-Native Space Intelligence Operating System（PSIOS）**，以物理驱动的世界演化为基础，以神经网络驱动的认知能力为核心，以自主决策为目标。

## 第二章 Physics-Native Intelligence：一种新的智能范式

### 2.1 从模型驱动到智能驱动

航天器自主控制技术始终围绕动力学模型展开。但未来任务——自主交会、在轨维修、空间机器人协同——不再是简单的轨迹跟踪，而是需要系统持续理解环境、分析任务、预测风险并自主调整行为。

AI 方法常将神经网络作为动力学模型的替代者，希望用大量数据学习状态演化。这种思路在地面自动驾驶中效果较好，但在航天领域存在局限：**轨道运动不是完全未知的复杂过程，而是受经典力学严格约束的。** 重新学习已明确的轨道动力学，增加训练成本也削弱了可解释性和安全性。

未来太空智能的发展方向不应是在模型驱动和数据驱动之间做选择，而应重新定义二者之间的关系。

### 2.2 模型驱动与数据驱动的职责重构

模型驱动与数据驱动分别解决不同层次的问题：

- **轨道动力学模型**描述客观世界如何演化——来源于物理定律，不是经验数据，始终由物理模型负责
- **神经网络**关注的是动力学模型难以直接描述的信息：任务语义理解、风险评估、多目标利益权衡、未知目标行为预测、长期任务经验积累

因此，物理模型和神经网络承担的是不同职责，而不是相同职责的两种实现方式。物理模型保证世界演化符合客观规律，神经网络赋予系统理解和适应世界的能力。**神经网络不再学习物理，而是学习物理之外的内容；物理模型也不再只是训练约束，而成为智能系统持续运行的重要组成部分。**

### 2.3 Physics-Native Intelligence 的定义

Physics-Native Intelligence 不是一种新的神经网络结构，也不是一种新的控制算法，而是一种智能系统设计范式。其核心思想是：**物理规律不是智能系统的外部约束，而是智能产生过程的内部组成部分。**

传统 Physics-Informed Learning 将物理规律体现在损失函数、正则项或网络结构设计中，训练完成后物理模型不再参与系统运行。Physics-Native Intelligence 则不同：**物理模型始终作为运行时组件参与整个智能循环**，与神经网络共同完成状态演化、环境认知和任务推理。神经网络不会覆盖物理模型，而是在物理模型提供的确定性基础上学习未知知识。

### 2.4 智能的产生机制

智能不是某一个模型输出的结果，而是持续推理过程的产物：

1. 依据当前状态完成动力学传播，获得未来状态及物理约束
2. 智能模型结合环境信息、历史经验和任务目标，进行语义理解，分析风险和机会
3. 系统依据任务目标生成候选行为
4. 物理模型验证行为是否满足动力学、资源约束和安全约束
5. 不可行则重新优化，直到获得满足所有约束的最终方案

整个过程循环运行，随着环境变化和任务推进不断更新。**智能不是一次计算结果，而是一种持续运行的能力。**

### 2.5 四项基本原则

1. **物理优先（Physics First）**：所有状态演化必须遵循客观物理规律，任何学习算法不能突破动力学约束
2. **职责分离（Responsibility Separation）**：物理模型负责世界演化，神经网络负责环境认知和经验学习，职责明确，不相互替代
3. **持续推理（Continuous Reasoning）**：智能来源于持续循环的推理过程，而不是单次模型推断
4. **可扩展（Extensibility）**：系统能不断增加新的智能能力，而无需改变底层运行机制

## 第三章 Physics-Native Intelligence Engine 总体架构

### 3.1 PNIE 的系统定位

Physics-Native Intelligence Engine（PNIE）是 PSIOS 的核心智能引擎。它不是替代传统 GNC 系统，而是在已有导航、制导与控制基础上，为航天器提供持续运行的自主智能能力。

传统航天软件按功能划分为独立模块，任务复杂度增加时模块间容易形成大量耦合。PNIE 不围绕具体任务构建，而是围绕智能能力的产生机制构建。导航、交会、补给等只是不同的能力表现，共享同一套运行框架。

在系统层次中，PNIE 位于任务系统与飞行控制系统之间：上层提目标，下层执行控制，PNIE 负责将任务目标转化为满足物理约束的自主行为。

### 3.2 为什么采用 Runtime 架构

传统模块（Module）适用于流程明确、功能边界清晰的软件，但不适合持续运行的智能系统。航天器需要持续进行轨道传播、环境认知、风险分析、任务评估和行为调整，这些能力贯穿整个任务生命周期。

PNIE 采用 **Runtime** 作为基本运行单元。Runtime 是一个持续运行的能力单元（Persistent Capability Unit），具有自己的生命周期、状态空间和更新机制。

优势：
- **持续性**：在整个任务过程中不断更新内部状态，形成连续的智能过程
- **独立性**：每种能力有明确职责和状态边界，通过统一状态空间交换信息，降低系统耦合
- **可扩展性**：新增能力只需增加新的 Runtime，不修改已有能力

### 3.3 Runtime 的统一定义

每个 Runtime 具备四项基本属性：

- **状态（State）**：维护自身负责的信息集合，在整个系统运行过程中持续存在
- **行为（Behavior）**：根据当前状态执行相应计算
- **接口（Interface）**：通过统一空间表示读取输入并写回输出
- **生命周期（Lifecycle）**：从系统启动运行到任务结束

### 3.4 PNIE 的运行组成

PNIE 由四类核心运行单元组成：

| Runtime | 职责 |
|---------|------|
| **Physics Runtime** | 维护航天器及空间环境的物理状态，完成轨道传播、姿态传播、可达域分析、约束计算及动力学一致性验证 |
| **Neural Runtime** | 完成动力学模型无法直接表达的智能认知：环境理解、目标行为预测、风险评估、经验学习、任务语义建模。不替代物理模型 |
| **Behavior Runtime** | 根据 Unified State Space 生成行为（Behavior），如自主导航、交会、巡视、补给等，根据当前任务动态调用 |
| **Memory Runtime** | 维护系统长期知识：任务历史、经验模型、地图信息、目标档案及长期学习结果 |

四类 Runtime 没有主从关系，在统一运行机制下协同工作。

### 3.5 Runtime 之间的协同机制

各 Runtime 不是按固定顺序执行，而是围绕统一状态持续协同运行：

1. Physics Runtime 根据观测和动力学模型更新物理状态，预测未来演化
2. Neural Runtime 读取最新状态，对环境、目标和任务进行认知分析
3. Behavior Runtime 依据任务目标和系统状态生成候选行为
4. Memory Runtime 持续维护长期知识

所有 Runtime 通过统一空间表示（Unified Space Representation）共享信息，不直接交换内部数据。随着新观测到来，整个系统立即进入下一轮更新，形成连续、自适应的智能运行闭环。

### 3.6 PNIE 总体架构

```

                     ┌──────────────────────────────┐
                     │      Mission Interface        │
                     └──────────────┬───────────────┘
                                    │
                                   ▼
                   ┌────────────────────────────────┐
                   │ Unified Space Representation    │
                   └────────────────────────────────┘
                         │            │
 ┌───────────┐  ┌───────────┐  ┌───────────┐  ┌───────────┐
 │Physics    │  │ Neural    │  │ Behavior  │  │ Memory    │
 │ Runtime   │  │ Runtime   │  │ Runtime   │  │ Runtime   │
```

三个基本特征：所有智能能力以 Runtime 形式持续运行；所有 Runtime 通过统一空间表示共享状态；飞行控制系统始终位于 PNIE 之外。

## 第四章 Physics Runtime——物理世界的持续运行层

### 4.1 设计目标

Physics Runtime 是 PNIE 中唯一负责维护客观物理世界的运行时组件。它的职责不是生成智能，而是保证整个智能系统始终建立在真实物理规律之上。

传统航天软件将动力学模型作为按需调用的算法工具，计算结束后便退出运行。PNIE 中，物理模型是一个持续运行的 Runtime，始终维护航天器及空间环境的物理状态，随时间不断更新。

设计目标：
1. 持续维护空间物理世界的数字化表示
2. 为其他 Runtime 提供一致、可信的物理状态
3. 保证所有智能行为满足动力学和工程约束

### 4.2 从轨道传播器到物理运行时

PNIE 不再将轨道传播视为单独算法，而是建立统一的 Physics Runtime，持续维护整个物理世界——不只是位置和速度，而是整个空间物理状态空间（Physical State Space），随时间不断演化。

### 4.3 Physics Runtime 的职责

**（1）动力学传播**
根据任务需求采用不同精度模型（二体、摄动、高精度轨道传播、姿态动力学），所有传播结果具有统一数据接口。

**（2）约束维护**
推进剂约束、推力约束、通信可见性约束、热控约束、姿态机动约束、电源约束——Physics Runtime 持续维护所有约束状态。

**（3）可达域分析**
持续维护可达域：可到达轨道集合、最小燃料轨迹、最小时间轨迹、安全轨迹空间。

**（4）资源演化**
推进剂消耗、电池荷电状态、太阳能输入、热平衡、执行机构寿命——这些具有明确物理规律的状态由 Physics Runtime 持续维护。

**（5）物理一致性验证**
验证所有候选行为是否满足动力学一致性、工程约束、安全边界和资源可行性。

### 4.4 Physics Runtime 的数据组织

Physics Runtime 维护完整的 Physical State：

| 状态类别 | 主要内容 |
|---------|---------|
| Orbit State | 轨道位置、速度、轨道根数 |
| Attitude State | 姿态四元数、角速度、姿态模式 |
| Dynamic Environment | 引力场、大气、太阳辐射压、第三天体 |
| Resource State | 推进剂、电量、热状态、执行机构状态 |
| Constraint State | 可见性、碰撞约束、姿态约束、推进约束 |

### 4.5 与 Neural Runtime 的关系

Physics Runtime 不学习、不预测任务语义，也不输出行为策略。Neural Runtime 不负责轨道传播、不维护资源状态、不修改动力学模型。二者唯一的联系是通过统一空间表示共享状态。

## 第五章 Neural Runtime——空间智能的认知运行层

### 5.1 设计目标

Neural Runtime 是 PNIE 中负责智能认知的持续运行层。在 Physics-Native 架构中，所有能够由物理模型准确描述的内容均由 Physics Runtime 负责维护；Neural Runtime 不再重复学习这些规律，而是关注物理模型无法直接表达的信息：目标意图、任务语义、环境风险、多目标协同、长期经验。

它维护的是 **Space Cognitive State（空间认知状态）**，而不是物理状态。

### 5.2 Neural Runtime 的职责边界

Neural Runtime **不**负责：轨道传播、动力学积分、姿态状态维护、执行机构控制量输出。

其职责集中在：
- **环境认知**：识别空间碎片密集区域、分析目标航天器运行状态、判断通信链路稳定性
- **目标理解**：理解目标的行为特征、任务属性及可能的运动趋势
- **风险推理**：多航天器资源竞争、通信中断、任务冲突等高层语义风险评估
- **经验学习**：积累不同轨道环境、任务模式和目标类型下的经验知识

Neural Runtime 的输出不是轨迹，而是**认知信息（Cognitive Information）**。

### 5.3 从状态预测到认知建模

传统数据驱动方法：给定当前状态，预测未来状态或轨迹。这种方法隐含假设神经网络需要学习整个世界的演化。

在航天系统中，世界演化已由轨道动力学严格定义。真正未知的不是状态，而是状态所蕴含的意义。例如，两颗卫星具有几乎相同的相对轨道状态，但一颗在执行自主交会，另一颗在进行碰撞规避——从动力学角度看轨迹相似，但从任务语义角度行为目的完全不同。

因此，Neural Runtime 不关注"状态是什么"，而关注"状态意味着什么"。

### 5.4 Space Cognitive State

| 认知类别 | 主要内容 |
|---------|---------|
| Environment Understanding | 空间环境理解、目标分布、潜在风险 |
| Mission Understanding | 当前任务阶段、任务完成度、目标优先级 |
| Intent Estimation | 目标行为意图、未来动作趋势 |
| Risk Assessment | 碰撞风险、资源风险、任务风险 |
| Experience Memory | 历史任务经验、策略评价、学习结果 |

Physics Runtime 告诉系统某目标未来十分钟的位置；Neural Runtime 判断该目标是否正在主动接近、是否可能改变轨道、是否值得优先交会。

### 5.5 持续学习机制

Neural Runtime 不采用一次训练、长期部署的固定模式，而采用持续学习机制，更新来自三个方面：任务反馈、环境变化、多智能体协同。这种持续学习不会改变 Physics Runtime 中的动力学模型，仅更新认知模型。

## 第六章 Unified State Space——统一状态空间

### 6.1 设计动机

传统航天控制系统将状态定义为动力学状态向量（位置、速度、姿态等），足以支撑轨道控制、姿态控制等传统任务。但自主交会过程中，系统不仅需要知道目标位置和速度，还需要理解目标的任务阶段、行为意图、环境风险和自身资源状况。

数据驱动方法引入了神经网络隐藏状态，但缺乏明确物理意义，难以与动力学状态建立统一关系。

因此需要一种新的状态描述方式，使物理状态和认知状态在同一框架下表达，共同参与系统推理。

### 6.2 Unified State Space 的定义

设系统在时刻 $t$ 的统一状态表示为：

$$S(t)=\{S_p, S_c, S_m, S_r\}$$

其中：
- $S_p$：Physical State（物理状态）
- $S_c$：Cognitive State（认知状态）
- $S_m$：Mission State（任务状态）
- $S_r$：Resource State（资源状态）

四类状态共同构成系统当前完整状态。USS 不要求不同状态采用相同数据类型，而是要求所有状态能在统一框架下组织、更新和访问。

### 6.3 状态空间组成

| 状态 | 描述 | 维护者 |
|------|------|--------|
| Physical State | 轨道状态、姿态状态、动力学环境、物理约束 | Physics Runtime |
| Cognitive State | 目标识别、风险评估、行为意图推断、环境语义、长期经验 | Neural Runtime |
| Mission State | 当前任务阶段、已完成任务、剩余目标、优先级变化、时间约束 | Behavior Runtime |
| Resource State | 推进剂、电源、热控、通信、计算资源、执行机构健康 | Physics Runtime |

### 6.4 状态之间的关系

USS 不是四类状态的简单拼接，它们之间存在持续耦合：

- Physical State 决定 Resource State 的变化（轨道机动→推进剂减少→电池变化→热环境变化）
- Cognitive State 影响 Mission State（识别到新风险→调整任务优先级）
- Mission State 又影响 Physical State（新任务目标→改变轨迹规划→影响未来轨道）
- Resource State 反过来限制 Mission State 的可执行范围（推进剂不足→重新规划任务）

**USS 本质上是一个动态耦合状态空间，整个智能系统的推理过程就是不断更新这一统一状态空间。**

### 6.5 Unified State Space 的运行机制

- Physics Runtime 更新 Physical State 和部分 Resource State
- Neural Runtime 更新 Cognitive State
- Behavior Runtime 根据任务目标持续更新 Mission State
- Memory Runtime 根据历史经验修正 Cognitive State 中的长期知识

系统中的任何能力均不直接依赖某一个 Runtime，而是依赖整个 Unified State Space。

### 6.6 Unified State Space 的意义

USS 解决了传统控制系统仅关注物理状态、现代神经网络仅维护隐藏状态所带来的割裂问题。未来太空具身智能需要同时回答四个问题：当前物理世界是什么状态？系统如何理解当前环境？当前任务处于什么阶段？当前资源是否支持下一步行动？

只有将这四类状态统一到同一状态空间，智能系统才能真正实现跨层次推理。

## 第七章 Unified Space Representation——统一空间表示

### 7.1 设计目标

USS 定义了智能系统需要维护的统一状态空间，USR 负责将这一抽象状态映射为可计算、可更新、可共享的软件对象。

PNIE 采用**状态驱动（State-Centric）** 而非模块驱动的运行模式。所有 Runtime 围绕 USR 协同工作：从 USR 读取当前状态，将更新写回 USR。

### 7.2 设计原则

1. **唯一状态源（Single Source of Truth）**：整个系统任意时刻只存在一个统一状态
2. **状态共享而非模块通信**：Runtime 之间不直接调用，通过 USR 协同
3. **异步运行**：各 Runtime 根据自身需求独立运行，USR 始终维护最新状态
4. **状态可追溯**：每一项状态记录来源、更新时间、产生模块及可信度
5. **统一状态访问接口**：访问轨道状态、任务状态还是认知状态均采用一致的数据接口

### 7.3 USR 的数据组织

```
Unified Space Representation
├── Physical State
│   ├── Orbit
│   ├── Attitude
│   ├── Environment
│   └── Constraint
├── Cognitive State
│   ├── Situation Awareness
│   ├── Intent Understanding
│   ├── Risk Assessment
│   └── Experience Context
├── Mission State
│   ├── Goal
│   ├── Current Behavior
│   ├── Progress
│   └── Priority
└── Resource State
    ├── Propellant
    ├── Power
    ├── Thermal
    └── Communication
```

### 7.4 Runtime 与 USR 的交互机制

工作流程：读取 USR 状态 → 完成自身职责更新 → 写回 USR → 等待下一周期。

各 Runtime 之间不存在直接依赖：Physics Runtime 完成轨道传播后仅更新 Physical State；Neural Runtime 自动读取最新 Physical State 并更新 Cognitive State；Behavior Runtime 依据最新 USS 生成行为。

### 7.5 Behavior Runtime 与 Capability Library

PNIE 将能力实现与行为生成分离：

- **Behavior Runtime** 负责回答：当前应该执行什么行为？
- **Capability Library** 负责回答：如何完成这一行为？

Capability Library 是一组可复用的智能能力（Navigation、Docking、Refueling 等），而不是持续运行的 Runtime。

### 7.6 USR 作为 Agent 的共享工作空间

Behavior Runtime 可以看作整个系统的 Agent。Physics Runtime、Neural Runtime 和 Memory Runtime 分别持续维护物理世界、认知世界和长期知识。USR 是连接物理推理、认知推理和行为生成的统一状态平台。

## 第八章 Behavior Runtime——行为生成机制

### 8.1 Behavior 的定义

Behavior Runtime 是 PNIE 中负责自主行为生成的核心运行层。它不直接输出控制指令，而是输出**行为（Behavior）**——系统在当前状态下应采取的行动方式（保持轨道、接近目标、建立编队、执行交会、实施补给或主动避碰）。

行为是一种高于控制、低于任务的智能表达。随着环境变化、任务推进和资源变化，Behavior Runtime 不断重新评估当前行为，行为本身也是系统状态的一部分。

### 8.2 行为生成的基本流程

每次运行包括四个阶段：

**（1）状态感知**：读取 USS，获取物理状态、认知状态、任务状态和资源状态

**（2）行为推理**：回答当前最重要的目标是什么？是否存在风险？资源是否支持？是否需要切换？结果不是控制命令，而是新的行为状态

**（3）能力选择**：根据当前行为，从 Capability Library 中选择对应能力（如"自主接近"→ Navigation Capability）

**（4）行为反馈**：Capability 完成执行后更新 USS，下一周期重新评估

### 8.3 行为层级

| 层级 | 关注 | 示例 |
|------|------|------|
| 战略行为 | 长期目标 | 轨道转移、建立补给链路、维护编队 |
| 战术行为 | 当前阶段 | 主动接近、姿态调整、交会等待、避碰 |
| 执行行为 | 具体能力 | 调用 Capability，监测执行结果 |

### 8.4 行为切换机制

行为切换不基于固定阈值或预设时间节点，而是基于 USS 的持续推理结果。例如自主补给任务的行为演化：

```
轨道保持 → 轨道转移 → 自主接近 → 姿态对准 → 自主交会 → 补给执行 → 安全撤离
```

每一次切换由当前 USS 决定，系统能够根据环境变化动态调整执行策略。

## 第九章 Behavior Graph——行为组织与任务演化

### 9.1 定义

Behavior Graph 不是有限状态机，也不是预定义任务流程，而是 **USS 上的行为组织模型**。它描述系统当前处于何种行为状态，以及不同状态之间如何演化。

Behavior Graph 是由行为节点（Behavior Node）和行为迁移（Behavior Transition）组成的有向图。

### 9.2 Behavior Node

每个节点是具有明确语义的行为状态，包含：行为名称、行为目标、激活条件、完成条件、推荐能力、安全约束。

例如"自主接近"节点：
- 行为目标：缩短与目标航天器的相对距离
- 激活条件：目标已识别且交会窗口可用
- 完成条件：进入预定接近区域
- 推荐能力：Navigation Capability
- 安全约束：满足最小安全距离和推进剂约束

### 9.3 Behavior Transition

迁移条件由 USS 实时状态联合决定：当前物理状态是否满足下一阶段动力学条件？认知状态是否确认目标环境安全？任务状态是否进入新阶段？资源状态是否支持？长期经验是否建议调整？所有条件共同满足时才触发迁移。

### 9.4 Behavior Graph 的运行机制

获取 USS → 评估当前 Behavior Node → 判断迁移条件 → 更新当前 Node → 调用对应 Capability → 等待下一周期。整个过程中 Behavior Graph 始终保持连续更新，不存在一次性规划后长期执行。

## 第十章 Physics–Neural Co-Reasoning——物理–神经协同推理

### 10.1 基本思想

PNIE 将物理推理与认知推理统一到同一个推理循环中：

| 角色 | 回答的问题 |
|------|-----------|
| Physics Runtime | 世界如何演化？ |
| Neural Runtime | 世界意味着什么？ |
| Behavior Runtime | 下一步应该采取什么行为？ |

### 10.2 推理循环

每个推理周期包括五个阶段：

1. **状态获取**：Behavior Runtime 从 USS 获取统一状态
2. **物理推理**：Physics Runtime 预测未来状态演化（轨道、相对运动、可达域、能源消耗、安全约束）
3. **认知推理**：Neural Runtime 进行目标行为预测、环境风险评估、任务上下文理解、历史经验检索、不确定性估计
4. **行为推理**：Behavior Runtime 综合两类推理结果，评估是否需要切换行为或调用新 Capability
5. **状态更新**：Capability 执行后，新状态写回 USR，下一周期重新开始

### 10.3 双预测机制

PNIE 同时维护两种预测：

- **物理预测**（Physics Runtime）：未来会发生什么——具有明确物理意义，满足动力学约束
- **认知预测**（Neural Runtime）：未来意味着什么——目标行为变化、风险演化趋势、任务成功概率

Behavior Runtime 在两类预测共同作用下形成最终行为。

### 10.4 推理一致性

三类一致性约束：
- **物理一致性**：所有行为必须满足轨道动力学和工程约束
- **任务一致性**：行为必须始终服务于当前 Mission State
- **认知一致性**：行为必须与当前环境认知保持一致（如 Neural Runtime 判断目标存在异常机动时，不应继续接近）

### 10.5 与传统 AI Agent 的区别

传统 LLM Agent：Observation → Reasoning → Action

PNIE 推理循环：Observation → Physical Reasoning → Cognitive Reasoning → Behavior Reasoning → Capability Execution → State Update

增加了物理推理层，行为生成与能力执行解耦。PNIE 是面向物理世界的 Agent，而不是面向文本世界的 Agent。

## 第十一章 Behavior Planner——自主行为规划

### 11.1 行为规划与轨迹规划

| 规划层次 | 回答的问题 | 示例 |
|---------|-----------|------|
| **行为规划（Behavior Planning）** | 当前系统应该采取什么行动？ | 保持轨道 / 主动接近 / 等待 / 避碰 |
| **轨迹规划（Trajectory Planning）** | 如何实现已确定的行为？ | 计算最优接近轨迹、推进剂消耗、姿态控制策略 |

### 11.2 Behavior Planner 总体架构

核心流程包括五个阶段：

1. **候选行为生成**：生成多个具有可行性的候选行为，而非唯一决策
2. **未来状态预测**：Physics Predictor 预测物理演化 + Neural Predictor 预测目标行为、环境变化和潜在风险
3. **物理约束筛选**：利用动力学、资源、安全约束剔除不可行候选
4. **行为评估**：综合任务收益、安全风险、资源消耗和长期影响进行统一评价
5. **行为确认**：选择最优行为，交由 Trajectory Planner 生成可执行轨迹

整个过程在每个规划周期持续重复。

### 11.3 候选行为生成机制

不采用端到端策略网络直接预测最终行为，而是首先生成多个候选行为，再通过物理约束和综合评价完成决策。原因：

1. 复杂航天任务通常不存在唯一正确答案（目标不可观测时，可等待可调整轨道可重新规划）
2. 候选行为提高可解释性——能说明为什么放弃其他方案
3. 使学习模型定位为行为生成器（Behavior Generator）而非最终决策器，未来模型替换不影响框架

### 11.4 Physics-First 的规划原则

PNIE 在规划开始阶段就利用物理模型约束整个搜索空间：

- 候选行为所需速度增量超过推进剂能力 → 无需继续评估
- 未来轨迹违反最小安全距离 → 直接剔除
- 轨道动力学预测显示无法形成交会窗口 → 不再进入后续规划

大量不合理方案在规划早期被排除，神经网络只需关注物理模型无法描述的不确定因素。

### 11.5 行为评价机制

在多维度综合分析：任务完成度、安全性、资源利用率、长期收益、鲁棒性。评价针对的是行为而不是轨迹。

### 11.6 与传统规划方法的区别

传统方法：以轨迹优化为中心，默认任务目标和行为模式已确定，缺乏自主选择行为的能力。

端到端方法：直接输出控制策略，缺乏显式物理约束，可解释性和安全性不足。

PNIE Behavior Planner 通过"候选行为生成—物理约束筛选—综合行为评价"三阶段机制，实现模型驱动与数据驱动的深度融合。

## 第十二章 Behavior Candidate Modeling——候选行为建模

### 12.1 行为表示的选择

| 表示方式 | 不适用的原因 |
|---------|-------------|
| 控制量（Control） | 属于 GNC 职责范围，Behavior Planner 应关注"做什么"而非"如何控制" |
| 轨迹（Trajectory） | 航天系统中轨迹由动力学决定，神经网络重复学习效率低且难以保证物理一致性 |
| 行为标签（Behavior Label） | 离散标签表达能力有限，无法描述接近距离、时间窗口等连续参数 |

### 12.2 Behavior Intent：候选行为的统一表示

PNIE 采用 Behavior Intent（行为意图）作为统一表示——不是控制指令也不是轨迹，而是对未来行为目标的参数化描述：

$$Intent = \{Goal, Constraint, TimeWindow, RiskPreference, Budget\}$$

Behavior Intent 描述系统希望达到的行为目标，而非达到目标的具体路径。Physics Runtime 根据 Intent 推导未来轨迹，Behavior Planner 根据预测结果评价优劣，Trajectory Planner 再生成最终控制轨迹。

### 12.3 Candidate Behavior Generator 的模型需求

1. 能生成多个候选行为（而非唯一输出）
2. 能处理连续参数（而非仅限于离散类别）
3. 能结合历史经验和当前环境进行推理
4. 能与 Physics Runtime 紧密协同

### 12.4 不同模型的适用性

| 模型 | 优势 | 局限 | 在 PNIE 中的定位 |
|------|------|------|-----------------|
| **Transformer** | 上下文建模能力强，成熟高效 | 判别式建模，不擅长描述不确定性 | 适合作为 Behavior Intent Generator |
| **Diffusion Model** | 擅长复杂分布，天然支持一对多生成 | 推理速度慢 | 适合离线学习，不适合实时 Planner |
| **Reinforcement Learning** | 能直接学习长期收益 | 奖励设计困难，缺乏可解释性 | 不建议作为 Candidate Generator 主体 |
| **World Model** | 能内部仿真预测未来状态 | 动力学完全由神经网络学习，忽略已知规律 | 不适合直接用作基础模型 |

### 12.5 Physics-guided World Model（长期目标）

本白皮书提出 Physics-guided World Model，其状态转移不完全由神经网络学习，而是采用物理模型与神经网络协同建模：

$$s_{t+1} = f_{physics}(s_t, a_t) + f_{neural}(s_t, a_t, c_t)$$

其中 $f_{physics}$ 表示轨道动力学、姿态动力学及环境模型；$f_{neural}$ 用于学习模型误差、未知机动及复杂环境语义；$c_t$ 表示 Neural Runtime 提供的环境认知信息。

Physics Runtime 负责构建可信的未来状态空间，Neural Runtime 仅补偿物理模型无法描述的不确定因素。

### 12.6 当前阶段的模型选择

第一代系统采用分层实现方案：
- Physics Runtime 负责未来轨道传播与约束预测
- Neural Runtime 采用 Transformer 对 Behavior Intent 建模
- Behavior Planner 在 Physics Runtime 提供的未来状态空间内完成候选行为搜索与评价

**Transformer 不是 PNIE 的最终形态，而是第一代工程实现中最合适的建模工具；Physics-guided World Model 才是未来持续演化的目标。**

## 第十三章 总结与展望

### 13.1 总结

随着空间基础设施发展，航天任务正从"单星执行"向"多航天器自主协同"演进。现有方法大多面向地面场景，其核心假设是通过大规模数据学习环境规律。而空间环境具有高度可建模、强物理约束和高安全可靠性的特点。

PNIE 的核心思想不是构建一个更大的神经网络，而是重新组织物理模型、数据驱动模型与自主规划之间的关系：

- **Physics Runtime** 维护物理世界并预测未来
- **Neural Runtime** 理解环境语义和不确定因素
- **Behavior Runtime** 完成自主行为规划
- **Memory Runtime** 积累长期经验

四个 Runtime 共同维护 Unified State Space，形成完整的自主运行闭环。与传统端到端系统不同，PNIE 将神经网络定位为智能规划过程中的认知增强模块，整个系统遵循 Physics First、Neural Enhancement、Behavior Planning 的设计原则。

Behavior Planner 的规划对象不是控制量或轨迹，而是 Behavior Intent。系统生成多个候选行为意图，结合物理预测、环境理解和综合评价完成行为选择，再由轨迹规划与控制模块完成具体执行。

### 13.2 展望

1. **完善 Behavior Planner 核心算法**：候选行为的生成、搜索与优化机制，特别是连续状态空间中的高效行为搜索方法
2. **逐步建立 Physics-guided World Model**：将 Physics Runtime 与 Neural Runtime 深度融合，构建兼具物理一致性与环境泛化能力的新一代世界模型
3. **建立大规模数字孪生空间环境**：覆盖轨道动力学、空间环境、多智能体交互和在轨任务的高可信仿真平台
4. **从单智能体向群体智能演进**：PNIE 的长期目标是成为支撑未来空间自主智能体、空间机器人及空间基础设施运行的统一智能底座

---

*人工智能的发展正在推动航天技术进入自主智能时代。未来真正的空间具身智能，不应仅依赖越来越大的神经网络，也不能完全依赖传统物理模型，而应建立在二者深度融合的基础之上。*
